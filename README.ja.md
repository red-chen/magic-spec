# Magic Spec

**AIコーディングエージェントのための仕様駆動開発スキル集。**

[English](./README.md) | [中文](./README.zh-CN.md) | 日本語

構造化されたワークフローで、曖昧なアイデアをリリース可能な機能に変換します：探索 → 提案 → 実装 → レビュー。すべてのステップが追跡可能、テスト可能、そして自己省察的です。

```
 探索             提案                実装              レビュー
┌─────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ アイデア  │───▶│ proposal.md  │───▶│ RED-GREEN-   │───▶│ コードレビュー│
│ 分析     │    │ specs/       │    │ REFACTOR     │    │ セキュリティ  │
│ 調査     │    │ design.md    │    │ 自己省察     │    │ 監査         │
│          │    │ test-design  │    │ 検証         │    │              │
│          │    │ tasks.md     │    │ アーカイブ    │    │              │
└─────────┘    └──────────────┘    └──────────────┘    └──────────────┘
```

## なぜ仕様駆動なのか？

AIコーディングエージェントは強力ですが、構造がなければ一貫性のない結果を生み出します。テストをスキップし、要件を忘れ、設計から逸脱し、証拠なしに「完了」と宣言します。

magic-specは**6つのスキル**で各段階に規律を強制します：

| 問題 | magic-specの解決策 |
|------|-------------------|
| 未知のコードやレガシーコードを盲目的に触る | `/magic-codebase-recon` が変更前にアーキテクチャ、ホットスポット、爆発半径、脆弱性スコアをマッピング |
| 間違ったものを構築 | `/magic-explore` がコミット前に調査 |
| 曖昧な要件 | `/magic-proposal` がGiven/When/Thenシナリオ付きの正式な仕様を作成 |
| テストのスキップ | `/magic-proposal` が実装前にテストを設計 |
| 計画からの逸脱 | `/magic-apply` が各タスク後にdesign.mdに対して自己省察 |
| 「動くはず」という主張 | `/magic-apply` が各チェックボックスに新鮮な検証証拠を要求 |
| セキュリティ問題の見落とし | `/magic-security-review` が12フェーズCSO級監査を実行 |
| 浅いコードレビュー | `/magic-code-review` が並列エキスパートレビュアーを配置 |

## スキル一覧

### `/magic-codebase-recon` — コードベース偵察と安全な変更インテリジェンス

未知のコードベースやレガシーシステムに触れる前に、全体を把握します。5つの分析軸をカバーした `brief.md` を出力します：

- **アーキテクチャ考古学** — レイヤー構造、統合ポイント、過去のアーキテクチャ転換、そしてgit履歴とソースコメントから発掘した既知の技術的負債
- **Gitホットスポット分析** — 高チャーンファイル、暗黙的な協調変更カップリング、バグ吸引モジュール、休眠ファイル
- **依存関係追跡と爆発半径評価** — ファンイン/ファンアウト分析、循環依存検出、任意ターゲットモジュールの3層爆発半径（直接 → 推移 → 実行時）
- **脆弱性スコアリング** — チャーン・カップリング・爆発半径・テストギャップの加重スコア → 5段階リスクバンド（🟢 STABLE から ⛔ CRITICAL）
- **安全な変更戦略** — リスク順の変更シーケンスと、各危険モジュールへの戦略提案（ストラングラーフィグ、特性化テスト、隔離レイヤー、フィーチャーフラグ、並列実行、段階的抽出）

読み取り専用。すべての発見は `brief.md` に出力します。実装変更は `/magic-proposal` と `/magic-apply` で行います。

### `/magic-explore` — 深層探索と発見

コードを一行も書く前に、アイデアの調査、コード分析、アプローチの比較、根本原因の追跡を行います。

**4つの探索モード：**
- **アイデア探索** — 曖昧なアイデアを明確な理解に変換
- **コードベース調査** — アーキテクチャのマッピング、データフローの追跡、パターンの発見
- **根本原因分析** — 証拠チェーンと仮説検証による体系的デバッグ
- **アプローチ比較** — 具体的なトレードオフで2-3の選択肢を比較

成果物ではなくインサイトを生成。理解が結晶化したら `/magic-proposal` に遷移します。

### `/magic-proposal` — 構造化された変更提案

理解を5つの成果物を含む完全な変更提案に変換します：

```
magic-spec/changes/<変更名>/
├── proposal.md        なぜ・何を — スコープ、機能、影響
├── specs/             何が変わるか — デルタ仕様（ADDED/MODIFIED/REMOVED）
├── design.md          どうやって — アーキテクチャ決定、主要ファイル、トレードオフ
├── test-design.md     どう検証するか — 仕様からテストへのマッピング、テストケース
└── tasks.md           手順 — 順序付き実装チェックリスト
```

**主要機能：**
- **デルタ仕様** — システム全体ではなく変更のみを記述（既存プロジェクトに対応）
- **Given/When/Thenシナリオ** — すべての要件にテスト可能なシナリオ
- **仕様-テストマッピング** — コードを書く前にすべての要件を具体的なテストケースにマッピング
- **依存関係グラフ** — 成果物を順序通りに作成：proposal → specs ←→ design → test-design → tasks

### `/magic-apply` — 仕様駆動の実装

TDD規律と継続的な自己省察でタスクを実行します：

```
各タスクについて：
  1. タスクコンテキスト + test-design.mdを読む
  2. 失敗するテストを書く（RED）
  3. 最小限の実装（GREEN）
  4. リファクタリング（REFACTOR）
  5. 証拠で検証
  6. 自己省察：設計適合？仕様充足？テストカバレッジ？
  7. 完了マーク（省察通過後のみ）
```

**自己省察は任意ではありません。** 各タスク後、エージェントは立ち止まって確認します：
- これはdesign.mdと一致しているか？
- すべての仕様シナリオがテストでカバーされているか？
- test-design.mdのテストケースが実際に実装されているか？
- 手を抜いていないか？

すべてのタスク完了後、**最終自己省察**がアーカイブ前に変更全体を監査します。

### `/magic-security-review` — セキュリティ態勢監査

12フェーズのCSO級自動セキュリティ監査：

| フェーズ | 検査内容 |
|---------|---------|
| 0 | アーキテクチャメンタルモデル + 技術スタック検出 |
| 1 | 攻撃対象面の調査 |
| 2 | シークレット考古学（git履歴） |
| 3 | 依存関係サプライチェーン |
| 4 | CI/CDパイプラインセキュリティ |
| 5 | インフラストラクチャシャドウサーフェス |
| 6 | Webhookと統合の監査 |
| 7 | LLM & AIセキュリティ |
| 8 | OWASP Top 10 |
| 9 | STRIDE脅威モデリング |
| 10 | 偽陽性フィルタリング + アクティブ検証 |
| 11 | データ分類 |

**2つのモード：** デイリー（8/10信頼度、ゼロノイズ）とコンプリヘンシブ（2/10信頼度、月次ディープスキャン）。

### `/magic-code-review` — マルチエキスパートコードレビュー

2段階レビュー + 並列エキスパート分析：

**ステージ1：仕様適合性** — 要求されたものを構築したか？仕様に対する要件ごとの検証。

**ステージ2：コード品質** — 構築したものは洗練されているか？アーキテクチャ、テスト、パフォーマンス、セキュリティ、保守性。

**エキスパートパネル**（並列）：テストエキスパート、セキュリティエキスパート、パフォーマンスエキスパート、保守性エキスパート — 発見事項をクロスバリデーションし重複排除。

## インストール（Claude Code）

### 1. リポジトリをクローン

```bash
git clone https://github.com/anthropics/magic-spec.git
```

### 2. プロジェクトにスキルを追加

スキルをプロジェクトの `.claude/skills/` ディレクトリにコピーまたはシンボリックリンク：

```bash
# プロジェクトルートから
mkdir -p .claude/skills

# 方法A：シンボリックリンク（推奨 — pull時に自動更新）
ln -s /path/to/magic-spec/skills/magic-codebase-recon .claude/skills/magic-codebase-recon
ln -s /path/to/magic-spec/skills/magic-explore .claude/skills/magic-explore
ln -s /path/to/magic-spec/skills/magic-proposal .claude/skills/magic-proposal
ln -s /path/to/magic-spec/skills/magic-apply .claude/skills/magic-apply
ln -s /path/to/magic-spec/skills/magic-security-review .claude/skills/magic-security-review
ln -s /path/to/magic-spec/skills/magic-code-review .claude/skills/magic-code-review

# 方法B：コピー
cp -r /path/to/magic-spec/skills/* .claude/skills/
```

### 3. CLAUDE.mdにスキルルーティングを追加（推奨）

プロジェクトの `CLAUDE.md` に以下を追加して自動スキル呼び出しを有効化：

```markdown
## Skill routing

ユーザーのリクエストが利用可能なスキルに一致する場合、最初のアクションとして呼び出す：

- コードベースへの初回接触、レガシー改修、「これを触ると何が壊れる？」→ magic-codebase-reconを呼び出す
- 機能アイデア、ブレインストーミング、「Xはどう動くのか」→ magic-exploreを呼び出す
- 「Xを構築」「機能を追加」「計画を立てて」→ magic-proposalを呼び出す
- 「実装して」「計画を実行」「コーディング開始」→ magic-applyを呼び出す
- セキュリティ監査、「脆弱性をチェック」→ magic-security-reviewを呼び出す
- コードレビュー、「変更をレビュー」→ magic-code-reviewを呼び出す
```

## 使用例（Claude Code）

### 例1：新機能 — フルワークフロー

```
あなた: アプリにダークモードを追加したい

Claude: [/magic-exploreを呼び出し]
  → 現在のスタイリングシステム、テーマアーキテクチャ、ユーザー設定ストレージを探索
  → 3つのアプローチとトレードオフを提案
  → CSSカスタムプロパティアプローチを推奨
  → 信頼度：8/10

あなた: 良さそうだね、それで行こう

Claude: [/magic-proposal add-dark-modeを呼び出し]
  → magic-spec/changes/add-dark-mode/ を作成
  → proposal.md を作成（なぜダークモードが必要か、何が変わるか）
  → specs/theme-switching.md を作成（ADDEDの要件とシナリオ）
  → design.md を作成（CSSカスタムプロパティ、ThemeProvider、localStorage）
  → test-design.md を作成（8つのテストケースを要件にマッピング）
  → tasks.md を作成（4グループ、14タスク）
  → 「進む前に成果物をレビューしてください。」

あなた: 承認、構築開始

Claude: [/magic-apply add-dark-modeを呼び出し]
  → タスク1.1：ThemeProviderの失敗テストを作成 → RED
  → タスク1.1：ThemeProviderを実装 → GREEN
  → タスク1.1：リファクタリング → クリーンアップ
  → タスク1.1：自己省察 → 設計適合 ✓、仕様カバレッジ ✓
  → タスク1.2：トグルの失敗テストを作成...（続く）
  → ...
  → 最終自己省察：全14タスク検証済み、設計一致 ✓、仕様カバレッジ ✓
  → 完了

あなた: マージ前にコードをレビューして

Claude: [/magic-code-review --spec add-dark-modeを呼び出し]
  → ステージ1：仕様適合 — すべての要件充足 ✓
  → ステージ2：コード品質 — 1件重要（エラーバウンダリ欠如）、2件軽微
  → 判定：修正必要 — 重要な問題1件
```

### 例2：クイックセキュリティチェック

```
あなた: /magic-security-review --diff

Claude: [現在のブランチの変更に対して12フェーズ監査を実行]
  → フェーズ0：Next.js + TypeScriptスタックを検出
  → フェーズ2：git履歴にシークレット漏洩なし ✓
  → フェーズ7：ユーザー入力がシステムプロンプトに流入を検出（重大、9/10）
  → フェーズ8：認証エンドポイントにレート制限なし（高、8/10）
  → 2件の発見、偽陽性0件
```

### 例3：修正前に調査

```
あなた: 最新デプロイ後にログインページが壊れた

Claude: [根本原因分析モードで/magic-exploreを呼び出し]
  → フェーズ1：エラーログを読み、認証フローを追跡
  → フェーズ2：動作する例を見つけ、壊れたコードと比較
  → フェーズ3：仮説 — 依存関係の更新でセッショントークンの形式が変更された
  → 検証：@auth/core 5.xがトークンのシリアライゼーションを変更
  → 信頼度：9/10、修正提案の準備完了

あなた: 修正して

Claude: [/magic-proposal fix-auth-tokenを呼び出し]
  → ターゲットを絞った提案を作成、1仕様、3タスク
  → test-designがリグレッションテストを具体的な障害点にマッピング
  ...
```

## アーキテクチャ

### 変更ライフサイクル

```
計画                        実装                           完了
────                        ────                           ────
/magic-explore              /magic-apply                   /magic-code-review
     │                           │                              │
     ▼                           ▼                              ▼
/magic-proposal             タスクごとのループ：            仕様適合性
     │                      RED → GREEN →                  コード品質
     ▼                      REFACTOR →                     エキスパートパネル
 proposal.md                検証 →                              │
 specs/                     自己省察                            ▼
 design.md                       │                         発見 + 修正
 test-design.md                  ▼
 tasks.md                   最終自己省察
                                 │
                                 ▼
                            アーカイブ
                            changes/archive/
```

### デルタ仕様

システム全体を記述する従来の仕様とは異なり、magic-specは**デルタ仕様**を使用します — 変更部分のみを記述：

```markdown
## ADDED 要件
以前存在しなかった新しい動作。

## MODIFIED 要件
変更された動作（差分だけでなく、更新後の完全な動作を含む）。

## REMOVED 要件
廃止される動作（移行ガイダンスを含む）。
```

これにより、既存システムを常に変更し続ける実際のプロジェクトで仕様が実用的になります。

### 自己省察メカニズム

magic-applyは2つのレベルの内省を含みます：

**タスクごとの省察**（各タスク完了後）：
```
タスク 1.1 省察
  設計適合：    ✓ design.mdの決定に一致
  仕様充足：    ✓ すべてのシナリオカバー済み
  テストカバレッジ：✓ test-design.mdの全ケース作成済み
  信頼度：      8/10
  → 完了マーク
```

**最終省察**（全タスク完了後）：
```
最終省察：add-dark-mode
═══════════════════════
タスク：        14/14（全て検証済み：はい）
設計一致：      ✓
仕様カバレッジ：✓
テストカバレッジ：✓
リグレッション：✓ なし
総合：          完了
```
